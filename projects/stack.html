Autonomous Navigation & Control Stack

System-level autonomy architecture separating estimation, planning, smoothing, sampling, and control into independently evolvable subsystems.

Overview

This project defines the full autonomy pipeline from sensing to drivetrain command dispatch. Each subsystem owns a strict contract and boundary so algorithm updates can be introduced without destabilizing adjacent modules. The architecture is built around deterministic execution order, enabling repeatable real-time behavior under iterative development.

System Architecture

Sensors
↓
State Estimation (EKF)
↓
Global Planning (A*)
↓
Path Processing (RDP + Smoothing)
↓
Arc-Length Sampling
↓
Motion Profiling (S-curve)
↓
Geometric Tracking (HPP)
↓
Drivetrain Interface

Data flow advances through explicit artifacts (pose, path, sampled reference, command), while control flow remains scheduler-driven. Keeping those concerns separate reduces hidden coupling and keeps timing behavior predictable.

Design Principles

  • Interface-first modular design: each stage publishes/consumes explicit contracts rather than shared internal state.
  • Deterministic scheduling assumptions: execution order is stable and bounded for repeatable runtime behavior.
  • Explicit coordinate-frame ownership: frame semantics are assigned per module to prevent transform ambiguity.
  • Unit-consistent data handoff: subsystem boundaries enforce predictable units to avoid silent scale errors.
  • Config-driven tuning (parameter structs): tuning moves through configuration layers, not algorithm rewrites.

These rules reduced integration regressions by containing change impact. Teams could iterate individual modules without forcing cross-stack revalidation on every tuning cycle.

What I Architected

  • Defined module boundaries and data contracts across subsystems.
  • Established trajectory lifecycle: path → smoothed path → sampled trajectory → control reference.
  • Structured configuration layers to isolate tuning from logic.
  • Designed predictable execution order for 100 Hz loop compatibility.
  • Coordinated subsystem evolution without cross-module breakage.

Integration Strategy

  • Estimator runs independently and publishes pose.
  • Planner consumes pose and publishes path.
  • Controller consumes trajectory sample and pose.
  • Drivetrain abstracts motor outputs.

This separation keeps subsystem dependencies one-directional and explicit. Lower coupling means planner/controller iterations do not require estimator rewrites, and actuator changes remain localized behind drivetrain interfaces.

Results & Validation

  • Reduced subsystem integration regressions through stable contracts.
  • Improved ability to tune controller behavior without touching estimator internals.
  • Stable runtime latency under iterative development.
  • Deterministic behavior across repeated runs.
  • Path execution consistency improved through stable interfaces and repeatable tuning.
  • Latency budget: designed for deterministic 100 Hz operation (10 ms cycle target).
